# F02: Harvester Completion — Expansion and Steady State

## Overview

**Feature:** Harvester Completion
**Problem:** F01 establishes the core loop on Ezra only. Selah is not covered, Ezra's native improvement system still runs in parallel (duplication), there is no operational visibility dashboard, no generic code health scanner, and no mechanism for the loop to reflect on its own effectiveness.
**Goal:** Add the remaining capabilities that bring Harvester to steady state: a web UI for operational visibility, Selah onboarding with its theological guardrails, a generic code health scanner applicable across all repos, a cross-repo pattern scanner that enables self-annealing, and removal of Ezra's now-redundant native improvement system.

## Context

F02 opens after F01 has been running stably for at least 30 days with confirmed end-to-end cycles. The 30-day stability gate is a hard dependency on F02-S05 (remove Ezra native system). The other stories in F02 can begin as soon as F01's M1 exit criteria are met (three merged cycles, continuous uptime).

The cross-repo pattern scanner (F02-S04) is the self-annealing mechanism from the Karpathy autoresearch analogy: it is the loop reflecting on its own results and proposing improvements to the loop itself. It requires a corpus of at least 30 closed issues before it activates — plan to start it 60-90 days after F01 completes.

## Non-Goals

- Voice of Repentance integration (future feature, after VoR repo is substantive)
- Multi-tenant support, SaaS productization
- Any AI Labor Solutions client-facing workflows
- GitHub Actions-based runner (subscription auth stays throughout)
- Public repo automation

## Dependencies

- F01 complete and stable (all six stories shipped, three end-to-end cycles confirmed)
- F02-S05 additionally requires 30 consecutive days of Harvester stability
- F02-S04 additionally requires 30+ closed issues across all configured repos
- F02-S02 requires Selah repo to have initial `CLAUDE.md` and guarded path structure

## Success Metrics

- All three personal repos (Ezra, Selah, VoR when ready) receive at least one useful finding per month
- Zero guarded-path changes merged to Selah without Troy's explicit review
- Web UI at `localhost:8500` provides complete operational visibility in under 30 seconds of checking
- Ezra's native improvement system fully removed; no duplication
- First cross-repo pattern finding produced and actioned

## Open Design Decisions

- **Selah CLAUDE.md.** The Selah repo needs its own `CLAUDE.md` before F02-S02 can be completed. This is a human-authored document, not generated by Harvester. It should cover Selah's theological constraints more explicitly than Ezra's operational conventions. Drafting it is a prerequisite task.
- **VoR inclusion timing.** Voice of Repentance may or may not be substantive enough to include in Harvester's watch list by the time F02 is underway. Include it if the repo has meaningful code or content; defer otherwise.
- **Pattern scanner corpus gating.** The 30-issue corpus gate for F02-S04 may be reached quickly or slowly depending on scanner cadence and merge rate. Do not force-start the pattern scanner before the gate is met — sparse data produces noise, not signal.
